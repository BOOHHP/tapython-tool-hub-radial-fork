# 网站与工具优化建议

## 测试观察

本次在 `http://localhost:5174/` 进行轻量人工走查，覆盖工具库首页、工具详情页“安装方式”和投稿/审核工作台。

已观察到的正向体验：

- 首页能快速看到工具数量、审核状态、版本快照和部署模式。
- 工具卡片展示了风险等级、版本、作者、挂载点和 UE 兼容版本。
- 详情页顶部直接提供 Markdown 和 ZIP 入口。
- “安装方式”页签同时覆盖 AI 助手、CLI 和 ZIP 三种安装方式。
- ZIP 区域展示了 sha256 和大小，便于人工审计。
- 投稿页已经把提交、校验、审核发布、版本不可变说明放在页面顶部。

已观察到的问题：

- 打开详情页时浏览器 console 出现 Ant Design 警告：`Descriptions` 的 `span` 合计与 `column` 不匹配，需要修正布局字段跨度。
- “安装方式”页签信息密度较高，AI 提示词、CLI 命令、ZIP 信息和手动安装步骤都在同一屏连续堆叠，首次用户需要阅读较多文本。
- CLI 安装区只显示当前平台的一条命令，Windows 命令隐藏在代码判断中；跨平台用户不容易确认自己应该复制哪条。
- AI 提示词中的 hub 地址在前端页面里使用 `http://localhost:5174`，而实际 API 服务是 `http://127.0.0.1:8787`。如果 Vite 没有代理对应 API，复制给 AI 后可能走错服务边界。
- 手动发布入口在“投稿”页内，审核和发布按钮在审核队列里；对“我现在要发布一个版本”的维护者来说，发布动作不够显眼。
- 审核队列里失败原因展示较长，缺少按字段折叠或复制错误详情的操作。
- 手动 ZIP 安装步骤没有内置“已完成”勾选状态，用户完成离线安装时缺少可追踪清单。

## 整改建议

### P0：修正正确性和信任问题

1. 修复详情页 `Descriptions` span 警告。
2. 明确前端生成安装提示词时的 hub base 来源：
   - API-backed 开发环境建议使用 `VITE_API_BASE_URL` 或 API 返回的 canonical hub base。
   - 如果使用前端域名，需要确认 Vite 代理 `/api`、`/downloads` 和 `/install` 都可用。
3. 在 CLI 和 ZIP 区域增加“复制命令”和“复制 sha256”的成功/失败反馈，剪贴板失败时显示可选中文本。

### P1：简化人类安装决策

1. 将“安装方式”改成更明确的三段式选择器：
   - 我想让 AI 帮我安装。
   - 我想用终端安装。
   - 我想手动下载 ZIP。
2. 每种方式只默认展开最少必要操作，更多细节放入“查看完整步骤”。
3. CLI 区域增加平台切换：macOS/Linux、Windows PowerShell、仅下载 CLI 包。
4. ZIP 区域增加完整清单：manifest、README、tool.md、MenuConfig、Python、UI JSON 是否包含。
5. 手动安装步骤增加可勾选清单，便于用户边做边核对。

### P1：强化发布工作台

1. 将“投稿”标签文案扩展为“投稿/发布”，降低维护者寻找成本。
2. 在投稿页顶部加入两个入口按钮：
   - 提交新工具。
   - 发布新版本。
3. 对已发布版本不可变的错误增加行动建议：自动提示“请把 version 改为 x.y.z+1 后重新提交”。
4. 审核队列增加筛选：待审核、已通过、已拒绝、校验失败。
5. 校验失败详情支持复制，便于维护者粘贴给 AI 助手或提交人修复。

### P2：让 CLI 更适合人类和 AI 共用

1. 增加 `tapython-tool-hub doctor --hub <url> --project <path>`，一次性检查 hub、TAPython 目录、MenuConfig 可写性和 CLI 版本。
2. `plan` 输出增加“下一步人类该做什么”的短句，减少用户读 JSON 或长文本的负担。
3. `download` 完成后输出本地 manifest、ZIP 路径和 sha256 校验结果。
4. `install --dry-run` 输出可保存为审计文件，例如 `--report install-plan.json`。
5. `uninstall` 前展示将删除的文件、将恢复的 MenuConfig 项和备份位置。

## 建议的产品文案调整

当前：`安装前需要预览写入文件和 MenuConfig 合并项`

建议：`先预览，再安装`

说明：副标题保留完整要求，例如“安装前请确认目标项目、写入文件、覆盖风险和 MenuConfig 变化”。

当前：`方式二：终端 CLI 安装`

建议：`终端安装，适合可运行命令的项目环境`

说明：用户先判断是否适合自己，再复制命令。

当前：`方式三：ZIP 包安装`

建议：`下载 ZIP，适合离线或手动审计`

说明：直接点明场景。

## 后续验证建议

- 对安装方式页签做一次 1366x768、1920x1080 和移动宽度截图，确认代码块不溢出。
- 用无 CLI 的新 shell 测试安装脚本，确认 PATH 提示清楚。
- 用真实 TAPython 测试项目执行一次 CLI install/uninstall 闭环。
- 用重复版本提交验证“已发布版本不可变”提示是否足够清楚。
- 用校验失败投稿验证审核队列的错误可读性。
